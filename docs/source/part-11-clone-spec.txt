Executive summary
Acuity Scheduling in the evidence set is a multi-surface scheduling platform with three major layers:
1. Public Scheduler
2. Client Self-Service
3. Admin Back Office

Additionally, the product includes:
- availability engine
- clients CRM
- configurable public scheduling page
- service catalog and forms
- money surfaces
- communications
- integrations and sync
- reports
- permissions shell
- host-style billing shell
- responsive mobile web parity

Several areas remain under-evidenced and must be treated conservatively:
- contributor permission granularity
- enterprise-only operational UI
- live processor-connected payment flows
- post-connect integration states
- group classes / class series lifecycle
- native app scope

Most important implementation implications
- Booking and self-service are tightly coupled to scheduling limits, forms, and communications.
- Payments, subscriptions, and invoices cannot be modeled faithfully without processor-state gating.
- Communications split into three UX models:
  - template-based client email
  - plan-gated SMS reminder editing
  - toggle-based admin alerts
- Calendar sync and external automations belong in an adapter layer.
- Mobile is responsive web parity, not a separate native-product baseline.
- Billing and Manage Users behave like host-shell surfaces and must remain architecturally distinct from regular scheduling-admin pages.

Critical architecture rules
1. Do not flatten Manage Users or Billing into ordinary scheduling pages.
2. Do not model 1-way ICS publication as true 2-way sync.
3. Do not create separate business logic for mobile.
4. Do not keep feature gating only in the frontend.
5. Do not let integrations own booking logic.
6. Appointment is the central aggregate.
7. Calendar sync must affect availability through explicit blockers/mapped busy events, not hidden side effects.
8. Host-shell account surfaces must remain separate from scheduling-admin surfaces.

Role matrix (practical summary)
- Anonymous / guest client: public booking
- Authenticated client: self-service, appointments, codes (evidence partial for non-empty states)
- Owner/Admin: core admin surfaces
- Contributor: implied, but granular permissions under-evidenced
- Enterprise admin: inferred from pricing/help only

Plan gating matrix (practical summary)
- Core scheduler and booking: broadly available
- Client email reminders/follow-ups: available broadly
- SMS reminders: paid-plan gate (Standard+ style)
- Packages/gift certificates/subscriptions: Standard+/Premium-style gate and processor dependency
- Payment processors/payment-at-booking settings: visible broadly, live behavior depends on active processor
- Scheduling invoices: shell visible, meaningful payment collection requires processor
- Advanced CSS / API credentials / custom developer surfaces: Premium / legacy Powerhouse gate
- Branding removal, HIPAA/BAA, multiple calendars, multiple staff time zones: pricing-evidenced higher-tier features
- Manage Users visible in current trial, but invitation is verification-gated
- Reports observed in current state
- Enterprise-only central user management remains inference/open assumption

Core business rules
- Single-service setups may deep-link directly to Date & Time
- Scheduling window is a combination of calendar availability and global scheduling limits
- Multiple times selection is supported before booking form
- Recurring booking is a separate modal branch
- Private appointment types must be hidden from public catalog and accessible by direct link only
- Forms with internal-use flag must not appear publicly
- Admin-only add-ons must not appear in self-booking flow
- Blank coupon expiration behaves as non-expiring
- Resources constrain capacity invisibly to clients
- Global scheduling limits affect booking/rescheduling/cancel/edit windows
- Confirmation page is the central self-service hub
- Packages / subscriptions / invoices depend on plan and processor state
- Communications are split into three systems: client email templates, SMS reminders, admin alerts
- Two-way sync is Cronofy-mediated
- One-way Calendar Sync is publication only
- Manage Users and Billing live in host-shell style surfaces
- Mobile differences are UX-level, not business-logic-level

Integration architecture rules
- Integrations are adapter-driven
- Webhooks are event-driven
- Sync providers are separate from one-way ICS publication
- API credentials and custom code surfaces belong to developer-tier/admin technical surfaces
- Provider states must be modeled explicitly: connected, disconnected, action-required, re-auth-required, error, never-connected

Mobile summary

- Client self-service mobile observed via login/confirmation-page modals
- Admin mobile partially observed across many modules
- 390x844 is the main validated viewport
- Native app is out of baseline scope

Important mobile frictions to preserve as explicit product decisions
- mobile coupon field absence vs desktop parity
- chip-style email input friction
- hidden admin modules behind hamburger + more navigation
- save buttons below the fold on long forms
- full-screen upgrade overlays
- reschedule modal scroll friction
- incomplete viewport matrix beyond 390x844

Cross-cutting technical rules
- Feature gating must evaluate role, plan, trial, dependency state, and verification state
- Booking, self-service, CRM, and notifications share appointment data
- Integrations subscribe to internal events; they do not own core scheduling logic
- Reports must tolerate sparse/empty datasets
- Mobile must reuse same routes, entities, and permissions as desktop
