Executive summary
Part 08 confirms that Acuity's integrations architecture is adapter-driven, not part of the booking core. It also proves that calendar synchronization must be modeled as two distinct capabilities:
1. Two-way sync through provider connections (via Cronofy)
2. One-way ICS/iCal publication

This distinction is critical for the clone.

Integrations inventory

Google Analytics
- Fields observed: Measurement ID, Secret Value
- Includes Anonymize IP checkbox
- Represents event-forwarding / analytics tagging rather than scheduling logic
- Should be modeled as account-level analytics config

Zoho Flow
- No native direct connect inside Acuity
- Surface is instructional and points to external pre-built flows
- This is a bridge, not a core native integration

Custom Sidebar Integration
- Injects custom HTML/widget into admin appointment sidebar
- Uses placeholders / variables from appointment and client context
- Should be modeled as a developer-tier HTML injection surface scoped to appointment sidebar rendering

Custom Conversion Tracking
- Injects HTML on confirmation page
- Uses dynamic variables / placeholders
- Should be treated as developer-tier confirmation-page tracking config

Webhooks
- Separate endpoint fields by event family
- Observed event classes include:
  - new appointment
  - rescheduled
  - canceled
  - catch-all update
  - order completed
- This implies event-driven integration architecture and per-event endpoint configuration

API Credentials
- Surface shows User ID and API Key
- Includes reset action and docs link
- Explicitly gated by legacy Powerhouse / premium developer gating
- Should be modeled separately from webhook surface and custom code surfaces

Zapier
Observed workflow cards:
- Google Contacts
- Salesforce
- Infusionsoft (empty/no workflows found)
- Google Sheets

Implication:
- Zapier is treated as workflow bridge inventory, not as an internal workflow builder
- Some integrations have ready-made recipes, some are empty-state informational surfaces

Calendar Sync inventory

Two-way sync providers observed
- Google Calendar
- iCloud
- Outlook Office 365
- Outlook Exchange
- Outlook.com

Common behavior observed in sync provider pages
- Connect button per provider
- Language implies appointments from Acuity go into external calendar
- External events can block Acuity availability
- Provider connection is mediated via Cronofy

iCloud-specific note
- iCloud flow references app-specific password

One-way Calendar Sync
- Separate dedicated page
- Publishes iCal/ICS outward
- Does NOT provide reverse availability blocking from subscriber calendars
- Must never be modeled as equivalent to two-way sync

Critical architecture rules derived from Part 08
1. Two-way sync and one-way ICS publication are separate modules.
2. Sync belongs in an adapter layer, not in booking-core logic.
3. External provider states should be modeled explicitly.
4. Webhooks are event subscribers/publishers, not booking workflow owners.
5. API credentials, Webhooks, Custom Sidebar, and Conversion Tracking are developer surfaces, not generic user settings.
6. Cronofy is the observed mediation layer for two-way calendar sync providers.

Observed business rules
- API and some developer functions are premium / legacy Powerhouse gated.
- Cronofy mediates two-way calendar sync.
- One-way sync is publication only.
- External bridges such as Zapier and Zoho Flow depend on the external service for actual automation behavior.
- Post-connect states were not fully validated because real external accounts were not connected.
